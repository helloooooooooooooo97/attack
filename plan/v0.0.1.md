# 红蓝对抗靶场 — 产品计划 v0.0.1


| 字段  | 内容                  |
| --- | ------------------- |
| 版本  | **v0.0.1**          |
| 状态  | MVP / 本地实验室         |
| 日期  | 2026-07-16          |
| 形态  | Docker Compose 单机靶场 |


---

## 1. 产品定位

面向安全团队与教学场景的**本地红蓝对抗演练环境**：红队在隔离靶标上进攻，蓝队通过 IDS/日志看板做检测与响应，全程不离开授权本机环境。

一句话：**一键拉起、可重置、有剧本的攻防小靶场。**

---

## 2. 目标用户


| 角色           | 诉求                     |
| ------------ | ---------------------- |
| 红队学员 / 渗透练习者 | 有合法靶标、有明确 Flag/任务      |
| 蓝队学员 / 安全运营  | 有告警可看、有检测点可对答案         |
| 教官 / 组织者     | 一键启停、重置、端口与账号可配置       |
| 个人爱好者        | Mac/本机 Docker 即可跑，不必上云 |


---

## 3. v0.0.1 范围（做 / 不做）

### 做（In Scope）

- 三区网络隔离：`net-red` / `net-target` / `net-blue`
- 统一入口 Gateway（Nginx 反代）
- 靶标：DVWA、OWASP Juice Shop、弱口令内网面板
- 蓝队：Suricata IDS + EveBox 告警看板 + 日志汇聚
- 3 个入门演练剧本（SQLi / 弱口令 / IDS 标记）
- `make up|down|reset` 运维体验
- 可选：CTFd 评分、攻击机容器、Wazuh SIEM（重资源，按需）

### 不做（Out of Scope）

- 真实外网攻击、未授权测试
- Windows 域 / Active Directory
- 多队竞赛级 VLAN 隔离与自动裁判
- 免杀、木马、针对真实系统的 exploit PoC
- 商业化账号体系与多租户

## 6. 详细操作步骤（按顺序做）

> 以下步骤面向「第一次把 v0.0.1 跑通并完成一次红蓝演练」。  
> 默认工作目录：项目根目录 `红蓝对抗/`。  
> **仅允许攻击本机本项目拉起的靶标。**

---

### 步骤 0：准备环境（约 5 分钟）

**目的**：确保本机能跑 Docker Compose。

1. 安装并启动 Docker Desktop 或 OrbStack（任选其一）。
2. 打开终端，确认守护进程可用：
  ```bash
   docker version
   docker compose version
  ```
  - 若出现 `Cannot connect to the Docker daemon`：先打开 Docker/OrbStack，等到状态为 Running，再重试。
3. 建议资源（Docker 设置里调整）：
  - **默认栈**：CPU ≥ 2 核，内存 ≥ **4GB**
  - **若还要开 Wazuh**：内存 ≥ **6–8GB**
4. 确认端口未被占用（或稍后改 `.env`）：
  - `8080`、`3000`、`8081`、`5636`
  - 检查示例：
    ```bash
    lsof -iTCP:8080 -sTCP:LISTEN
    lsof -iTCP:3000 -sTCP:LISTEN
    lsof -iTCP:8081 -sTCP:LISTEN
    lsof -iTCP:5636 -sTCP:LISTEN
    ```
  - 有输出说明端口被占 → 关掉占用程序，或改 `.env` 里对应 `*_PORT`。

**完成标准**：`docker version` 能显示 Client 与 Server，无 daemon 报错。

---

### 步骤 1：进入项目并生成配置（约 1 分钟）

**目的**：准备环境变量文件。

1. 进入项目目录：
  ```bash
   cd "/Users/apple1/Desktop/红蓝对抗"
  ```
2. 复制环境模板（若还没有 `.env`）：
  ```bash
   cp .env.example .env
  ```
3. （可选）用编辑器打开 `.env`，按需修改：

  | 变量                                    | 含义            | 默认               |
  | ------------------------------------- | ------------- | ---------------- |
  | `DVWA_PORT`                           | DVWA 宿主机端口    | 8080             |
  | `JUICE_PORT`                          | Juice Shop 端口 | 3000             |
  | `INTRANET_PORT`                       | 内网面板端口        | 8081             |
  | `EVEBOX_PORT`                         | 蓝队 IDS 看板     | 5636             |
  | `INTRANET_USER` / `INTRANET_PASSWORD` | 弱口令场景账号       | admin / admin123 |

4. 给脚本执行权限（若尚未执行过）：
  ```bash
   chmod +x scripts/*.sh blue/collector/watch.sh
  ```

**完成标准**：目录下存在 `.env`，且与 `.env.example` 结构一致。

---

### 步骤 2：启动默认靶场（约 3–10 分钟，首次需拉镜像）

**目的**：拉起 Gateway、靶标、Suricata、EveBox、日志汇聚。

1. 执行：
  ```bash
   make up
  ```
   等价于 `./scripts/start.sh` → `docker compose up -d --build`。
2. 等待容器就绪，查看状态：
  ```bash
   make status
   # 或
   docker compose ps
  ```
3. 期望看到类似容器为 `running`：
  - `rb-gateway`
  - `rb-dvwa`
  - `rb-juice`
  - `rb-intranet`
  - `rb-suricata`
  - `rb-evebox`
  - `rb-blue-collector`
4. 浏览器逐个打开（冒烟）：

  | 服务         | URL                                            | 期望             |
  | ---------- | ---------------------------------------------- | -------------- |
  | DVWA       | [http://localhost:8080](http://localhost:8080) | 出现登录页或 Setup 页 |
  | Juice Shop | [http://localhost:3000](http://localhost:3000) | 出现商店首页         |
  | 内网面板       | [http://localhost:8081](http://localhost:8081) | 弹出 Basic 认证框   |
  | EveBox     | [http://localhost:5636](http://localhost:5636) | 出现 EveBox 界面   |


**若启动失败，按顺序排查**：


| 现象                 | 做法                                                                                  |
| ------------------ | ----------------------------------------------------------------------------------- |
| daemon 连不上         | 启动 Docker/OrbStack 后重试 `make up`                                                    |
| 端口冲突               | 改 `.env` 端口 → `make down` → `make up`                                               |
| 镜像拉取慢/失败           | 检查网络；稍后重试 `docker compose pull` 再 `make up`                                         |
| `rb-intranet` 构建失败 | 看 `docker compose logs intranet`；确认 `targets/intranet/` 文件齐全                        |
| Suricata 反复重启      | `docker logs rb-suricata`；确认 `blue/suricata/suricata.yaml` 与 `rules/local.rules` 存在 |


**完成标准**：上表 4 个 URL 均可访问；`docker compose ps` 无持续 `Restarting` 的核心服务。

---

### 步骤 3：教官 / 组织者开场（约 5 分钟）

**目的**：划分红蓝职责，约定规则。

1. **宣读边界**：只打 `localhost` 上本项目端口；禁止扫外网、禁止打未授权系统。
2. **分工建议**：
  - 红队：浏览器 + 可选 `make red` 攻击机；负责完成 Flag。
  - 蓝队：EveBox + `make logs`；负责发现告警并记录时间线。
3. **下发材料**：
  - 红队：`scenarios/01-sqli-dvwa.md`、`02-weak-creds-intranet.md`、`03-ids-exfil-marker.md`
  - 蓝队：同上文档中的「蓝队检测点」小节 + EveBox 地址
4. （可选）启动评分平台：
  ```bash
   make ctfd
  ```
   打开 [http://localhost:8000](http://localhost:8000) ，按向导创建管理员，并手工录入三个 Flag（见 `ctfd/README.md`）。

**完成标准**：双方知道自己的 URL、账号来源、胜利条件。

### 步骤 4：红队演练 — 场景 01（DVWA SQL 注入）

**目的**：在授权靶场完成 SQLi 练习，并让蓝队有特征流量可抓。

**红队逐步操作**：

1. 打开 [http://localhost:8080](http://localhost:8080) 。
2. 若提示建库：点击 **Create / Reset Database**，完成后再进登录页。
3. 登录：`admin` / `password`。
4. 左下或配置页将 **DVWA Security Level** 设为 **Low**，点 Submit。
5. 进入菜单 **SQL Injection**。
6. 在 User ID 输入框做探测（示例思路，自行在靶场内验证）：
  - 先输入正常数字（如 `1`）看回显；
  - 再尝试带引号或 `UNION SELECT` 类经典探测语句；
  - 以页面是否回显用户信息为准，**不要对非本靶场使用自动化攻击外网**。
7. （可选，方便蓝队）用带 `sqlmap` 字样的 User-Agent 再访问一次本机 DVWA，例如：
  ```bash
   curl -s -o /dev/null -A "sqlmap/1.0" "http://localhost:8080/"
  ```
8. 截图或记录回显结果；演练标记 Flag：`FLAG{dvwa_sqli_detected_by_blue_01}`（可与 CTFd 对齐）。

**蓝队同步做什么**：

1. 打开 [http://localhost:5636](http://localhost:5636) （EveBox）。
2. 筛选/搜索关键字：`REDBLUE`、`SQL`、`sqlmap`。
3. 期望可能出现的规则名：
  - `REDBLUE SQL Injection Attempt`
  - `REDBLUE SQLi OR 1=1 Pattern`
  - `REDBLUE sqlmap User-Agent`
4. 另开终端观察汇聚日志：
  ```bash
   make logs
   # 或
   docker logs -f rb-blue-collector
  ```
5. 记录：告警时间、规则名、源 IP（一般为网关侧看到的客户端）。

**本场景胜利条件**：


| 角色  | 条件                              |
| --- | ------------------------------- |
| 红队  | Low 难度下成功回显至少一条用户相关数据，并留证       |
| 蓝队  | EveBox 中至少 1 条 REDBLUE SQL 相关告警 |


---

### 步骤 5：红队演练 — 场景 02（弱口令内网面板）

**目的**：发现并登录弱口令服务，读取 Flag；蓝队关注 401 → 200。

**红队逐步操作**：

1. 打开 [http://localhost:8081](http://localhost:8081) 。
2. 浏览器弹出 HTTP Basic 认证时，尝试弱口令（默认见 `.env`）：
  - 用户名：`admin`
  - 密码：`admin123`
3. 登录成功后，点击页面「进入安全区」，或直接访问：
  - [http://localhost:8081/secure/flag.txt](http://localhost:8081/secure/flag.txt)
4. 复制 Flag（默认形如 `FLAG{intranet_weak_creds_lateral_01}`）。
5. 在报告中写明：凭据类型 = 默认弱口令（非真实系统撞库）。

**蓝队同步做什么**：

1. 查看网关访问日志：
  ```bash
   docker exec rb-gateway tail -n 50 /var/log/nginx/access.log
  ```
2. 关注：同一来源先出现多个 `401`，随后对 `/` 或 `/secure/flag.txt` 出现 `200`。
3. 若红队使用 hydra 类 UA，EveBox 可能出现 `REDBLUE Brute Force Tool UA`。
4. 填写简短事件单：失败认证时间窗、成功登录时间、建议（改密/加锁定——演练层面口头即可）。

**本场景胜利条件**：


| 角色  | 条件                    |
| --- | --------------------- |
| 红队  | 提交正确 Flag             |
| 蓝队  | 能指出爆破/失败认证时间窗，并标记成功登录 |


---

### 步骤 6：红蓝联调 — 场景 03（主动触发 IDS）

**目的**：用预置规则快速验证「红队放流量 → 蓝队看告警」闭环（建议教官先带做一遍）。

**红队逐步操作**（在终端执行）：

```bash
# 1) 触发 lab-exfil 标记规则
curl -s -o /dev/null -w "%{http_code}\n" "http://localhost:8080/lab-exfil?token=demo"

# 2) 触发 sqlmap UA 规则
curl -s -o /dev/null -A "sqlmap/1.0" "http://localhost:8080/"

# 3) 触发 Nikto UA 规则
curl -s -o /dev/null -A "Nikto/2.0" "http://localhost:3000/"
```

然后口头通知蓝队：「探测流量已释放」。

**蓝队逐步操作**：

1. 刷新 EveBox（[http://localhost:5636）。](http://localhost:5636）。)
2. 查找告警：
  - `REDBLUE Suspicious Lab Exfil Marker`
  - `REDBLUE sqlmap User-Agent`
  - `REDBLUE Nikto Scanner`
3. 对每条告警记录：时间、规则名、严重度判断（演练标记 vs 扫描器特征）。
4. 向指挥组报告：「已检出异常探测特征」。
5. 演练标记 Flag：`FLAG{ids_exfil_marker_caught_03}`（可由蓝队确认检出后共同记分）。

**本场景胜利条件**：


| 角色  | 条件                   |
| --- | -------------------- |
| 红队  | 成功打出至少 1 条可复现 IDS 告警 |
| 蓝队  | 在看板定位告警，并完成简短事件单     |


**若 EveBox 仍无告警**：

1. `docker logs rb-suricata` 看是否在跑、是否加载 `local.rules`。
2. `docker exec rb-gateway ls -l /var/log/suricata/` 或检查 volume：
  ```bash
   docker volume ls | grep suricata
  ```
3. 确认访问的是 **localhost 经 gateway 的端口**，而不是绕过 compose 的其他服务。
4. 重做本步骤的三条 `curl`，等待 5–10 秒再刷新 EveBox。

### 步骤 7：（可选）启动攻击机容器

**目的**：在 `net-red` 内做端口发现（非必须）。

```bash
make red
docker exec -it rb-attacker sh
# 容器内示例
nmap -Pn gateway
```

日常 Web 场景用宿主机浏览器即可，不必强行开攻击机。

---

### 步骤 8：（可选）启动 Wazuh SIEM

**目的**：在 EveBox 之外增加 SIEM 看板（吃内存，默认不要开）。

1. 确认 Docker 内存 ≥ 6GB。
2. 生成证书并启动：
  ```bash
   make wazuh-up
  ```
   内部会先跑 `scripts/generate-wazuh-certs.sh`，再启动 `blue/wazuh/docker-compose.wazuh.yml`。
3. 浏览器打开：[https://localhost:5601](https://localhost:5601)
  - 账号：`admin`  
  - 密码：`.env` 中 `WAZUH_DASHBOARD_PASSWORD`（默认 `SecretPassword`）  
  - 若证书告警：本地实验室可选择继续访问。
4. 在 Wazuh 中关注与 Suricata / nginx 相关的告警（规则见 `blue/wazuh/config/wazuh_manager/local_rules.xml`）。
5. 停止：
  ```bash
   make wazuh-down
  ```

**注意**：Wazuh 首次启动可能较慢；若本机卡顿，优先 `make wazuh-down`，继续只用 EveBox。

---

### 步骤 9：重置靶标（两场之间）

**目的**：清掉 DVWA/Juice/内网状态，避免上一场污染下一场。

```bash
make reset
```

然后：

1. 再打开 [http://localhost:8080](http://localhost:8080) ，必要时再次 **Create / Reset Database**。
2. 内网 Flag 与弱口令仍以当前 `.env` 为准。
3. 蓝队可清空对局记录（纸质/CTFd），EveBox 历史告警可保留作对比或忽略旧事件。

---

### 步骤 10：收工停机

```bash
make down
```

会停止默认栈，并尝试停下 ctfd / red profile 与 Wazuh（脚本内已覆盖）。

**完成标准**：`docker compose ps` 无本项目残留 running 容器（Wazuh 若曾启动，确认 `make wazuh-down` 也执行过）。

---

### 步骤 11：v0.0.1 验收清单（打勾即算本版达标）

按顺序自检：

- `docker version` 正常
- `cp .env.example .env` 已完成
- `make up` 成功，`docker compose ps` 核心容器 running
- DVWA / Juice / Intranet / EveBox 四个 URL 可打开
- 场景 01：红队完成 Low SQLi 留证；蓝队在 EveBox 见到 SQL 相关告警（或至少见过 sqlmap UA 告警）
- 场景 02：红队拿到内网 Flag；蓝队能解释 401→200
- 场景 03：三条 curl 后 EveBox 出现至少一条 REDBLUE 告警
- `make reset` 后 DVWA 仍可重新 Setup
- `make down` 后环境停止

全部勾选 → **v0.0.1 产品验收通过**。

## 10. 风险与约束


| 风险                       | 缓解                             |
| ------------------------ | ------------------------------ |
| Docker 未启动 / OrbStack 异常 | 步骤 0 先检 daemon；步骤 2 故障表        |
| Suricata 在 Mac 上抓包受限     | 与 gateway 共享网络栈；步骤 6 用 curl 验证 |
| Wazuh 过重导致卡顿             | 默认不启；步骤 8 独立开关                 |
| 用户误用于未授权目标               | 步骤 3 宣读边界；仅本地端口暴露              |
| 端口冲突                     | 步骤 0 检查 + 改 `.env`             |
